Environment Variables, Logging & Monitoring

How to secure your app with environment variables, log like a pro with structured JSON, monitor health, and measure test coverage.

🧠 Core Concepts

These are the building blocks for securing, logging, and monitoring your application.

🔑

Environment Variables

Key-value pairs that live outside your code. They store config like ports, credentials, and feature flags. Read them via process.env.NAME in Node.js. Never hardcode secrets in source files.

📦

.env & dotenv

The .env file stores environment variables for local development. The dotenv npm package reads it and sets process.env values. Always add .env to .gitignore — never commit secrets!

🔒

GitHub Secrets

Encrypted values stored in your GitHub repo settings. Injected into workflows via ${{ secrets.NAME }}. Used for production credentials — server IP, SSH keys, auth passwords. Masked in logs (best-effort).

📋

Log Levels

Control what gets logged: debug (verbose details) → info (normal events) → warn (suspicious activity) → error (failures). Set LOG_LEVEL env var to filter output. Default: info.

📝

Structured Logging

Log as JSON instead of plain text. Each entry includes timestamp, app, level, and message. Makes it easy to search, filter, and analyze logs with tools like Kibana or Grafana.

🏢

Central Logger Pattern

All logging goes through one class (logger.js). The rest of the app calls log.info(), log.error(), etc. To switch to an external service later, you only change the _write() method — zero changes to the rest of the code.

❤️‍🩹

Health Check Endpoints

A GET /health route that returns the app's status, uptime, and timestamp. No auth required — monitoring tools and load balancers hit this to check if the app is alive. If it fails, they can alert you or auto-restart.

📊

Code Coverage

Measures what percentage of your code is actually tested. Types: Statements (lines run), Branches (if/else paths), Functions (called), Lines. Jest generates HTML reports. Uploaded as GitHub Actions artifacts.

logger.js — The Central Logger

1 const LEVELS = { debug: 0, info: 1, warn: 2, error: 3 }; Map of log levels to numbers. Lower number = more verbose. Used to compare against the configured threshold.
2 const APP_NAME = require('../package.json').name; Reads your app name ("pipeline") from package.json. Added to every log entry so you can filter logs when multiple apps run on the same server.
3 class Logger { A single class handles ALL logging. This is the "central" part — every file imports the same instance.
4 constructor(options = {}) { Takes optional config. In practice, the level comes from LOG_LEVEL env var.
5 this.level = LEVELS[level] || LEVELS.info; Set the minimum level. Default is "info" — debug messages are hidden unless you explicitly enable them.
6 info(message, meta) { this._log('info', message, meta); } Public method. Call log.info('something happened', { id: 1 }) anywhere in the app. Same pattern for debug, warn, error.
7 _log(level, message, meta) { Internal method that all public methods call. Checks the level, builds the entry, writes it.
8 if (LEVELS[level] < this.level) return; KEY: If the message level is below the threshold, skip it entirely. This is how LOG_LEVEL=warn hides info and debug messages.
9 const entry = { timestamp, app: APP_NAME, level, message }; Build a structured JSON object. Every entry has the same shape — easy to parse, search, and analyze.
10 if (meta !== undefined) entry.meta = meta; Optional extra data (like request URL, item id). Only added if provided — keeps logs clean.
11 _write(level, entry) { THE MAGIC: This is the only method you change when switching to Kibana, Grafana, Datadog, etc. Right now it writes to stdout/stderr.
12 process.stdout.write(JSON.stringify(entry) + '\n'); Writes JSON to stdout. Errors go to stderr. pm2 captures both into log files you can read with pm2 logs.
13 module.exports = new Logger(); Singleton pattern. Exports a single instance. Every file that requires logger.js gets the SAME object — consistent config everywhere.

app.js — Auth with Env Vars & Structured Errors

1 const log = require('./logger'); Import the singleton logger. Every file calls log.info(), log.warn() instead of console.log.
2 require('dotenv').config(); Loads .env file into process.env. Only needed locally — in production, pm2's ecosystem.config.js sets the variables instead.
3 app.use((req, res, next) => { Request logger middleware. Runs on EVERY request before any route handler.
4 log.info(`${req.method} ${req.url}`); Logs method and URL at "info" level. Output: {"app":"pipeline","level":"info","message":"GET /api/items"}
5 const AUTH_USER = process.env.AUTH_USER || 'admin'; Read username from environment. The || 'admin' fallback is only for local dev convenience. Production MUST set the secret.
6 const AUTH_PASS = process.env.AUTH_PASS || 'changeme'; Same for password. The fallback 'changeme' screams "replace me in production" — that's the point.
7 log.warn('Auth failed: no credentials', { url }); Auth failures logged as "warn" — suspicious but not fatal. Includes URL but never the password (security!).
8 res.status(401).json({ error: 'AUTH_INVALID', message: 'Incorrect username...' }); Structured error response. The error field is a machine-readable code. The message is human-readable and shown to the user in the UI.
9 app.get('/health', (req, res) => { Health check endpoint. NO auth required — monitoring tools need open access. Returns status, uptime, timestamp.
10 log.info('Item created', { id: item.id, name: item.name }); Successful operations logged at "info" level with relevant metadata. Helps track what happened and when.

server.js — Startup with Debug Logging

1 const log = require('./logger'); Same logger instance. All startup logs go through the central logger too.
2 log.debug('Starting server', { dirname, cwd, port, authUser: 'set' }); Debug-level startup info. Hidden by default (LOG_LEVEL=info). Shows paths and whether secrets are configured — never their values!
3 log.info(`Server running on http://localhost:${PORT}`); The famous "server started" message. At info level so it always shows. Tells you which port — useful when running multiple apps.
