CI/CD Pipelines with GitHub Actions

Everything you need to know about automating your code from push to production — explained step by step with animations.

🧠 Core Concepts

Before diving into the pipelines, let's understand the building blocks.

🔄

CI — Continuous Integration

Every time you push code, tests run automatically. This catches bugs immediately — before anyone else sees them. Think of it as a spell-checker for code.

🚀

CD — Continuous Deployment

If all tests pass, the code is deployed to a server automatically. No manual SSH, no copying files. Push → Tests → Live.

📄

Workflow File (YAML)

A .yml file inside .github/workflows/ that tells GitHub Actions what to do and when to do it. Like a recipe — step by step instructions.

⚡

Trigger (on:)

The event that starts the pipeline. Examples: push to a branch, creating a PR, on a schedule. Without a trigger, nothing runs.

💼

Job

A group of steps that run on the same machine. Jobs run in parallel by default. Use needs: to create a dependency chain (run after another job).

👣

Step

A single action inside a job — either uses: (a pre-built action) or run: (a shell command). Steps always run sequentially.

🖥️

Runner

A temporary virtual machine that GitHub creates to run your job. It's brand new each time — empty filesystem, no history. Think of it as a disposable computer.

🔑

Secrets

Encrypted values stored in GitHub settings (like passwords, SSH keys, server IPs). They're injected as environment variables at runtime. Never hardcode secrets in workflow files.

📦

npm ci

Clean Install — installs exact versions from package-lock.json. Faster and more reliable than npm install. Always use in CI. --omit=dev skips test tools.

📡

rsync

A tool that copies files from one machine to another. Only sends files that changed — much faster than copying everything. Uses SSH for secure transfer.

🔄

pm2

A process manager for Node.js. Keeps your app running forever — restarts it if it crashes, manages logs, handles multiple apps on different ports.

🛡️

Branch Protection

Rules on GitHub that prevent direct pushes to master. Code can only enter master through a Pull Request that passes all checks. The gatekeeper.

📖 Pipeline Code — Line by Line

Click any line to highlight it. Every line explained in the simplest possible way.

test-branch.yml — The Main Pipeline

1 name: Test Branch Pipeline A label for this workflow — shows up in GitHub's Actions tab so you can find it.
2 on: The trigger — defines WHEN this pipeline runs. Like saying "run this recipe when..."
3 push: ...someone pushes code (git push). Other options: pull_request, schedule, workflow_dispatch.
4 branches: [test] Only the test branch. Pushing to master or any other branch won't trigger this.
5 jobs: Start of the jobs section — the actual work the pipeline does.
6 test: First job named "test". This is an ID — use any name you want.
7 runs-on: ubuntu-latest Use a fresh Ubuntu virtual machine. It's free and fast. Options: ubuntu-latest, macos-latest, windows-latest.
8 steps: List of actions to run — like a to-do list executed top-to-bottom.
9 - uses: actions/checkout@v4 Download your repo code onto the runner. Without this, the VM is empty — it has no files!
10 - uses: actions/setup-node@v4 Install Node.js on the runner. Like installing Node on a new computer.
11 with: { node-version: '20', cache: 'npm' } Use Node v20. cache: npm saves downloaded packages between runs — makes installs faster.
12 - run: npm ci Install ALL dependencies (express, jest, supertest). npm ci = clean install from lock file. Always use in CI.
13 - run: npm test Run all tests. If ANY fails → job fails → deploy is blocked. ✅ = safe to deploy.
14 deploy-test: Second job — deploy to the server. Only runs if the test job passed.
15 needs: test KEY: Wait for the "test" job to finish first. Without this, deploy and test run at the same time!
16 if: success() Only run if ALL jobs in "needs" passed. If tests failed, skip this entire job.
17 env: { TEST_SERVER_HOST: ${{ secrets.TEST_SERVER_HOST }} } Read encrypted secrets from GitHub Settings. Your server IP, username, SSH key — all hidden.
18 mkdir -p ~/.ssh Create the .ssh folder on the runner. Fresh VM = nothing exists yet.
19 echo "$TEST_SERVER_KEY" > ~/.ssh/deploy_key Write the SSH private key from secrets into a file. This key unlocks the server door.
20 chmod 600 ~/.ssh/deploy_key Lock file permissions. SSH refuses keys that anyone can read — must be 600 (owner only).
21 ssh-keyscan -H $HOST >> known_hosts Remember the server's fingerprint. Without this, SSH asks "Are you sure?" and hangs forever.
22 ssh ... "mkdir -p /var/www/pipeline" Create app folder on the server (if first deploy). -p = no error if already exists.
23 rsync -avz --exclude='node_modules' ... Copy changed files to server. -a=archive, -v=verbose, -z=compress. Skips node_modules (server installs its own).
24 ssh ... "cd /var/www/pipeline && npm ci --omit=dev" On the server: install only production deps (express). --omit=dev skips jest — server doesn't test.
25 (pm2 delete pipeline || true) && pm2 start ... Stop old app, start fresh. || true = don't crash if app wasn't running. Clean slate every deploy.

📋 Quick Reference Cheat Sheet

Git Commands

git add . # Stage ALL files (incl. dotfiles) git add * # ⚠️ Skips .github/, .gitignore git commit -m "msg" # Save staged changes git push origin # Push to GitHub git status # See what's staged/changed

npm Commands

npm ci # Clean install (all deps) npm ci --omit=dev # Production only (no jest) npm test # Run test suite npm start # Start the server npm run bump:patch # 1.0.0 → 1.0.1 npm run bump:minor # 1.0.0 → 1.1.0 npm run bump:major # 1.0.0 → 2.0.0

pm2 Commands

pm2 list # Show all running apps pm2 logs app # See app's console output pm2 start file.js # Start an app pm2 delete app # Remove an app pm2 save # Remember current setup pm2 restart app # ⚠️ Keeps old env/cwd

SSH / rsync

ssh user@host # Connect to server ssh -i key user@h # Connect with key file chmod 600 keyfile # Set key permissions rsync -avz src dst # Copy changed files ssh-keyscan host # Get server fingerprint
